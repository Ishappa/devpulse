Why this project (Staff-Engineer justification)

You were given four candidate directions. Here's my ranking and the reasoning, because choosing the project well is itself an interview signal.

The winning move: pick a project whose hardest part is a system-design problem, not a UI problem. DevPulse's core is "aggregate two rate-limited, eventually-consistent data sources, compute a derived trend signal on a schedule, and serve it sub-second to authenticated users." Crucially, it satisfies every mandatory requirement organically — SSR (personalized feed), ISR (trending pages), SSG (marketing/about), infinite scroll (feed), real-time-ish (poll/revalidate), search+filter (tech radar), rate limiting (API protection + upstream budget), caching (the whole point). 01Project Name

DevPulse — with the tagline "Feel the pulse of what developers are building."

Memorable, product-shaped, domain-available pattern (e.g. devpulse.app), and it names the core value prop: a pulse (a moving signal over time), not a static list.

02One-Line Pitch

DevPulse aggregates GitHub and Hacker News in real time and computes a unified momentum score so engineers can see which technologies, repos, and ideas are gaining traction before they hit the mainstream.

03Problem Statement

Developers suffer from signal fragmentation. The information that predicts what's worth learning is scattered:

  • GitHub trending shows stars, but stars are a lagging, gameable vanity metric and the page has no history.
  • Hacker News shows discussion, but it's a firehose with no per-technology aggregation.
  • Neither tells you velocity — is this rising or already peaked? — which is the only thing that matters for "should I invest my learning time here."

DevPulse solves this by joining the two sources around entities (a repo, a language, a topic) and computing a time-decayed momentum score that blends star velocity, HN discussion volume, and sentiment. The output answers a real question: "What is gaining momentum right now, and is it accelerating?"

04Interview Story

This is the narrative you walk an interviewer through. It demonstrates that you think like an engineer who ships, not a tutorial-follower.

"I wanted to track emerging dev tools but the data lived in two places with hostile constraints — GitHub's API gives you 5,000 authenticated requests/hour, Hacker News' Firebase API is unauthenticated but you have to fan out one request per item. So the project became an exercise in respecting upstream rate limits while serving fresh data fast.

I built an ingestion layer that runs on a cron (Vercel Cron + a queue), normalizes both sources into a common entity model, and computes a momentum score with time decay. The read path never touches the upstream APIs — it reads from Postgres and a Redis cache, so a viral spike in my traffic can't blow my GitHub budget. That decoupling of write-path (ingest) from read-path (serve) is the core architectural decision, and it's exactly what you'd do at scale with Kafka and a worker fleet — I just did the single-node version with a cron and Upstash."

That story hits: rate-limit reasoning, read/write path separation, caching, derived data, and an explicit "here's how it scales" — all unprompted.

Schema (DDL sketch)

-- entities: the thing being tracked (a repo, an HN-surfaced topic, a language)
CREATE TABLE entities (
  id            BIGSERIAL PRIMARY KEY,
  type          entity_type NOT NULL,            -- enum: 'repo' | 'topic' | 'hn_story'
  source_key    TEXT NOT NULL,                   -- e.g. 'vercel/next.js' or hn item id
  name          TEXT NOT NULL,
  url           TEXT,
  description   TEXT,
  language      TEXT,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (type, source_key)                      -- idempotent upsert key
);

-- snapshots: append-only time series, one row per entity per ingest run
CREATE TABLE snapshots (
  id              BIGSERIAL PRIMARY KEY,
  entity_id       BIGINT NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
  captured_at     TIMESTAMPTZ NOT NULL,
  stars           INT,
  stars_delta     INT,                           -- vs previous snapshot (pre-computed)
  hn_points       INT,
  hn_comments     INT,
  momentum_score  NUMERIC(8,3) NOT NULL,         -- the derived signal
  UNIQUE (entity_id, captured_at)
);

CREATE TABLE users (
  id          BIGSERIAL PRIMARY KEY,
  github_id   BIGINT UNIQUE NOT NULL,
  email       TEXT,
  image_url   TEXT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE watches (                            -- a user's personalized watchlist
  id         BIGSERIAL PRIMARY KEY,
  user_id    BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  entity_id  BIGINT NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (user_id, entity_id)
);

Indexing strategy

IndexWhy
UNIQUE(entities.type, source_key)Upsert target for idempotent ingest; also the natural lookup key.
UNIQUE(snapshots.entity_id, captured_at)Prevents duplicate snapshots on queue retries (at-least-once delivery).
idx_snapshots_latest (entity_id, captured_at DESC)Powers "latest snapshot per entity" via DISTINCT ON — the hottest query.
idx_snapshots_score (captured_at DESC, momentum_score DESC)Powers the trending leaderboard for a given time bucket without a full scan.
idx_entities_language (language) + GIN on name/description via pg_trgmFiltering by language and fuzzy text search in the tech radar.
UNIQUE(watches.user_id, entity_id)Idempotent "watch" toggling; doubles as the lookup index.

Query optimization considerations

  • Latest-snapshot query uses SELECT DISTINCT ON (entity_id) ... ORDER BY entity_id, captured_at DESC — index-backed, no window-function sort over the whole table.
  • Pre-compute on write, not read: stars_delta and momentum_score are written by the ingest worker so the read path is a plain indexed select — push work to the cold path.
  • Snapshots grow unbounded → partition by month (PARTITION BY RANGE (captured_at)) or a rollup job that downsamples old data to daily. Documented as the 100k-user fix.
  • Connection pooling: serverless functions exhaust Postgres connections; route through Supabase's PgBouncer (transaction mode) or Prisma Data Proxy.

09State Management Strategy

The mental model: three kinds of state

State kindLives inExample
Server state (canonical, remote)RSC fetch + ISR cache; or RTK Query for client-fetched authed dataTrending list, entity snapshots, the user's watchlist
Client UI state (ephemeral, global)Redux Toolkit slicesRadar filter/sort config, comparison tray, command-palette open state, theme
Local component stateuseStateAn input's value, a dropdown's open/close

Why Redux Toolkit

  • The filter/sort config on /radar is read by the results grid, the active-filter chips, and the URL-sync effect — three distant subtrees. A shared store beats prop-drilling.
  • The "comparison tray" (pin entities to compare sparklines) persists across route changes — global, not page-local.
  • RTK Query gives me request dedup, caching, and invalidatesTags for the authed, non-ISR-cacheable views (watchlist mutations) — without hand-rolling fetch state.

Why NOT Context API

Context has no concept of selectors — any value change re-renders every consumer. The radar filter state changes on every keystroke; with Context the entire results grid would re-render per character. Redux's useSelector with referential equality (or createSelector memoization) lets only the affected components re-render. Context is for low-frequency, truly-global values (theme, locale) — and even those I'd colocate, not centralize.

Senior nuance: "Most of my 'state' is actually server state, and the App Router handles that with the fetch cache — so Redux stays small and only owns genuine client UI state. Reaching for Redux to hold server data is the anti-pattern I avoid."

Validation strategy

const SearchQuery = z.object({
  q:      z.string().trim().min(1).max(64).optional(),
  lang:   z.string().max(32).optional(),
  window: z.enum(['1d', '7d', '30d']).default('7d'),
  cursor: z.string().max(256).optional(),
  limit:  z.coerce.number().int().min(1).max(50).default(20),
});
// Parse at the boundary; the inferred type IS the handler's input type.
const parsed = SearchQuery.safeParse(searchParamsToObject(req));
if (!parsed.success) return json(400, { error: 'VALIDATION', issues: parsed.error.issues });
  • Cursor, not offset: stable under concurrent inserts and O(log n) via the indexed (score, id) tuple.
  • Consistent error envelope (error code + details) so the client can branch reliably.
  • Every external input (query, body, headers) crosses a Zod gate — never trust the client.

12MVP Roadmap — Week 1–2

Week 1 — Foundation & write path

  • Day 1–2: Scaffold Next 14 + TS + Tailwind. Supabase schema + migrations (Drizzle). Upstash Redis + QStash provisioned. CI (lint, typecheck, test) on GitHub Actions.
  • Day 3–4: Upstream clients (github.ts, hackernews.ts) with typed responses + retry/backoff. scoring.ts momentum algorithm — TDD'd, since it's pure.
  • Day 5–7: Ingest pipeline: /api/cron/ingest fans out to QStash → /api/jobs/ingest-entity normalizes + upserts snapshots idempotently. Verify with Vercel Cron on a preview deploy.

Week 2 — Read path & core UI

  • Day 8–9: /trending as ISR with generateStaticParams for top languages. Redis read cache on the leaderboard query. Loading/empty/error states.
  • Day 10–11: /radar search + filter via /api/search (Zod + cursor pagination + rate limit) with infinite scroll (Intersection Observer + RTK Query).
  • Day 12–13: NextAuth + GitHub OAuth. /feed SSR personalized. watches toggle with optimistic update + RTK Query tag invalidation.
  • Day 14: Entity detail page with snapshot sparkline (lightweight SVG, no heavy chart lib). Responsive pass + Lighthouse audit.
Definition of MVP done: a public user sees fresh trending data with search; an authed user has a personalized feed and watchlist; the read path never calls an upstream API; Lighthouse ≥95 on / and /trending.

15Security Measures

Authentication flow

GitHub OAuth via NextAuth. User clicks login → redirected to GitHub → consents → GitHub redirects back with a code → NextAuth exchanges code for token server-side → issues a signed, httpOnly, SameSite=Lax JWT session cookie. The OAuth access token is encrypted in the JWT, never exposed to client JS.

Authorization strategy

  • Route Handlers call auth() server-side; unauthenticated requests to /api/watches, /api/feed get 401 before any DB access.
  • Edge Middleware gates (app) routes — redirect to login if no session.
  • Resource-level checks: a user can only mutate their own watches (WHERE user_id = session.userId) — never trust an id from the body. Admin /ops gated by an allowlist role claim.

XSS prevention

  • React escapes by default; never use dangerouslySetInnerHTML on upstream data (repo descriptions can contain markup).
  • Strict Content-Security-Policy header (nonce-based for scripts) via middleware — defense in depth.

CSRF protection

  • NextAuth's built-in CSRF token on auth routes. SameSite=Lax cookies block cross-site form posts. State-changing endpoints are JSON-only + check Origin — a cross-site form can't set Content-Type: application/json.

Rate limiting

  • @upstash/ratelimit sliding window — per-IP on public endpoints (e.g. 30 req/10s on /api/search), per-user on authed ones. Returns 429 + Retry-After.
  • Separate upstream budget guard: the ingest worker tracks remaining GitHub quota in Redis and backs off — protects me from my own pipeline exhausting the API.

Input validation & secure API design

  • Zod at every boundary (covered in §10). Parameterized queries only (ORM) — no string-built SQL → no SQLi.
  • Secrets in Vercel env vars, never in the bundle. Cron + QStash endpoints verify a shared secret / signature so they can't be triggered by the public.
  • Security headers: Strict-Transport-Security, X-Content-Type-Options: nosniff, X-Frame-Options: DENY.

  • Test: the scoring algorithm (business-critical, pure, cheap to test), API contracts (envelope, status codes, auth boundaries), and the ingest idempotency guarantee.

20Interview Talking Points (per feature)

Ingestion pipeline (write path)

  • Why it exists: upstream APIs are rate-limited and slow; serving from them directly is fragile and blows the quota.
  • Alternatives: (a) fetch-on-request with caching — still risks thundering-herd cache stampedes on cold keys; (b) one big cron loop — times out at Vercel's function limit, no per-item retry.
  • Tradeoff: QStash gives at-least-once delivery → I pay with idempotent upserts (unique constraint) instead of exactly-once complexity.
  • How to explain: "It's the single-node version of a Kafka + worker-fleet ingestion pipeline. Same shape, different scale."

Momentum score (derived data)

  • Why: raw stars are a lagging vanity metric; velocity with time-decay answers the real question.
  • Alternatives: raw star count (gameable, no recency), simple delta (noisy). I use exponential time-decay weighting recent activity.
  • Tradeoff: computed on write (stale up to 15 min) vs. on read (fresh but expensive). I chose write — see scoring deep-dive.

Caching (Redis + ISR)

  • Why two layers: ISR caches rendered HTML at the edge (kills TTFB for everyone); Redis caches the query result for dynamic/authed paths ISR can't serve.
  • Tradeoff: staleness vs. freshness — tuned per route (15 min for trending, 60s for search results, on-demand revalidate for entity detail).
  • Invalidation: "there are only two hard problems…" — I use TTL + on-demand revalidatePath from the ingest worker rather than trying to be clever.

Auth & personalization

  • Why JWT strategy: stateless, no session-table read per request — lower latency, simpler scaling.
  • Tradeoff: can't instantly revoke a JWT server-side; acceptable for this app, and I'd add a short TTL + denylist if it mattered.

Search + infinite scroll

  • Why cursor pagination: OFFSET is O(n) and skips/dupes rows under concurrent inserts; cursors on an indexed tuple are stable and O(log n).
  • Why virtualization: rendering 1000 DOM rows tanks INP; virtual windowing keeps node count constant.
